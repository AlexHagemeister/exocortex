# ISSUES — friction log (open-issue index)

One file per issue in `.state/issues/` (`<YYYY-MM-DD>-<slug>.md`), frontmatter `title` / `status: open|resolved|wontfix` / `opened` / `rule` (what it implicates) / `resolved_by` (filled on close); body freeform, no ceremony — a flag that takes effort won't get filed. To file: create the file, regenerate this index. To close: flip `status`, fill `resolved_by`, regenerate. Resolved files stay on disk as greppable history; this index lists **open issues only** and is derived — never hand-maintained. The weekly digest surfaces open issues; the amend skill's cluster threshold triggers an amend proposal.

**Program defects go upstream too.** If the friction is in the shipped program itself — a skill, rule, or tooling bug any deployment would hit, not your local amendments or your data — file it on the program repo's public issue tracker as well (upstream URL per meta/DEPLOYMENT.md; `gh issue create`, or draft the text for the user to post). Write it in the detail you'd want on the receiving end: what happened, what you expected, the file and line, your environment. It is a public post — no vault content, no personal data, and the user approves before it goes up; unattended runs leave the draft in the local issue file for the digest to surface.
